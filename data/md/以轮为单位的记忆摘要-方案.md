### Implementation Plan, Task List and Thought in Chinese

#### Thought

1. **分析用户的新思路**：用户认为原方案虽然不动数据库，但代码里判断消息类型、寻找 ID 边界的逻辑“弯弯绕绕”容易变复杂，从而提议：新增一张轮次表（比如：`llm_turn`），将 `conversation_id` 和 `turn_id` 显式关联。
2. **可行性评估**：**这个思路非常清晰且可行性极高。**在系统设计中，一旦发现某两个实体在概念上是应该分离的（单条 Message 与 一整个对话 Turn），建立显式的表映射（显式建模）永远是维护成本最低、逻辑最直白的解法。同时，这种设计具有极强的未来延展性：
   - 以后如果要记录单轮对话产生消耗的 Token、耗时时长；
   - 或者给单次回答点赞、踩（用户反馈）；
   - 或者针对某单轮生成一个临时的小结，都可以挂在 `llm_turn` 这张表上。
3. **架构闭环设计**：
   - 如果新增 `llm_turn` 表代表一轮会话的主体，那么自然而然的，在 `llm_memory` 表中我们需要增加一个冗余字段 `turn_id` 来作外键。
   - 每当用户发起新 query（接口或入口截获时），我们生成一个 `turn_id` 并将其写入新建的 `llm_turn` 表，同时放进全局的 `RequestContext` 或 `Mono` 的上下文里。
   - 此次请求中途产生的所有消息（User输入、System提示词、Tool调用、Tool结果、Assistant回复），在被存入 `llm_memory` 的时候统一自动打上这个上下文里的 `turn_id` 标志。
4. **加载与压缩逻辑极大简化**：
   - **加载记忆**：要查最近 10 轮。SQL 就是 `SELECT turn_id FROM llm_turn WHERE conversation_id = ? ORDER BY id DESC LIMIT 10`。拿到这十个 `turn_id`，直接 `WHERE turn_id IN (...)` 查出所有的 message。极其简单优雅，不会出现边界截断的bug。
   - **压缩判断**：统计未压缩的轮数只需要 `SELECT COUNT(1) FROM llm_turn WHERE id > last_turn_id`（假设摘要表里保存的不再是最老的 message 主键，而是变成最老的 turn主键 `last_turn_id`）。如果大于阈值，就去拿到前几轮的 `turn_id` 集合，随后把挂载在它们底下的 message 统一查出来做压缩！

#### Implementation Plan (实施方案：显式轮次建模方案)

**1. 数据库模型调整 (Database Schema)**

- **新增表 `llm_turn`**：
  主键 `id` (自增/雪花)，`turn_id` (轮次唯一流水号，或直接服用主键)，`conversation_id` (会话ID)，`create_time` (创建时间)。
- **修改表 `llm_memory`**：
  补充字段 `turn_id`，使其作为外键映射到 `llm_turn`。
- **修改表 `llm_summary`**（可选但推荐）：
  将用于判断水位线的 `last_memory_id` 升级为 `last_turn_id`，表明目前已压缩的最新轮次的坐标。

**2. 代码链路生成机制 (Generation)**

- **上下游入口**：在用户每次发起交互的起点拦截处（或者是处理用户提问的第一步代码里），生成本次交互的 `turn_id` 并存库 `llm_turn`。
- **上下文透传**：将生成的 `turn_id` 同步赋值进 `RequestContext`。由于底层使用 `MemoryManager.saveMemory` 统一存储，此时从 Context 提取出 `turn_id`，给传入的所有的新消息（`List<Message>`）统一赋值，一并落入 `llm_memory` 表。

**3. 核心查询与摘要逻辑重构 (Query & Compress)**

- **统计轮数更直白**：[countNormalMessages](cci:1://file:///d:/Study/git/ai-study/base-agent/src/main/java/com/lyh/base/agent/memory/repository/MysqlMemoryRepository.java:51:4-60:5) 直接改为查 `llm_turn` 表的水位线后 Count 数量。
- **加载最老的 `toCompressTurns`**：计算要压缩多少轮时，从 `llm_turn` 表拿最早的 `N` 条记录获取对应的 `turn_id`，再去查 `llm_memory`。由于是精确根据ID IN查询，绝不会截断或漏掉任何 `tool_call` 数据！
- **更新水位线**：摘要重写完成后，这批被压缩的 turn 里面那个最大、最新的 `turn_id` 便成了 `LlmSummary` 标定的最新位点。

这种方案不仅抛却了“查找某条 message 猜测它的断点”这种隐式判断，同时让核心逻辑的代码行数大幅下降，变得异常清晰。可以说是一个典型的**空间换取高内聚逻辑**的好架构。
